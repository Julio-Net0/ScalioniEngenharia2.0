01. Camada de Domínio

O Coração do Software

A camada de domínio é a mais importante da aplicação. Nela residem as regras de negócio puras que ditam o comportamento do sistema.

Características Principais:

  • Totalmente isolada de frameworks ou bibliotecas (React, Next.js).
  • Independente de canais de comunicação (HTTP, Banco de Dados, LocalStorage).
  • Fácil portabilidade e altíssima testabilidade em milissegundos.

Padrão DDD (Domain-Driven Design)

Dividimos esta camada em peças fundamentais para manter a clareza e isolamento semântico:

- Entities (Identidade única)
- Value Objects (Imutáveis por valor)
- Domain Errors (Exceções de negócio)
- Repository Interfaces (Contratos de abstração)

Domínio: Value Object

Classe Preco

Um Value Object (Objeto de Valor) é um objeto imutável definido exclusivamente por seus atributos. Não possui ID. Seus valores determinam igualdade estrutural.

📂 src/core/domain/value-objects/Preco.ts

Autovalidação Imediata

Garante a integridade dos dados no construtor. É impossível ter um preço negativo em qualquer ponto do sistema.

export class Preco { constructor(public readonly valor: number) { if (valor < 0) { throw new PrecoNegativoError(); } } public equals(outro: Preco): boolean { return this.valor === outro.valor; } public formatarBRL(): string { return new Intl.NumberFormat('pt-BR', { style: 'currency', currency: 'BRL' }).format(this.valor); } }

02. Camada de Aplicação

Orquestração e Casos de Uso

A camada de aplicação traduz as interações do usuário em fluxos lógicos e sequências de passos.

Responsabilidades:

  • Orquestrar a recuperação de entidades a partir de repositórios.
  • Executar as regras de fluxo do negócio que não pertencem a uma entidade específica.
  • Garantir a integridade dos dados transitados na operação.

Caso de Uso (Use Case)

Cada fluxo de negócio é implementado em uma classe dedicada com responsabilidade única. A entrada de parâmetros é validada e a saída é tipada com objetos de domínio.

Aplicação: Caso de Uso

ObterPlantaDetalhadaUseCase

Este caso de uso busca uma planta a partir do slug e verifica se ela está ativa para exibição pública no catálogo.

📂 src/core/application/use-cases/ObterPlantaDetalhadaUseCase.ts

Injeção pelo Construtor

Depende da interface IPlantaRepository, respeitando o princípio de inversão de dependência (DIP).

export class ObterPlantaDetalhadaUseCase { constructor(private readonly repo: IPlantaRepository) {} public async execute(slug: string): Promise<Planta> { if (!slug) throw new Error("Slug é obrigatório"); const planta = await this.repo.getPlanta(slug); // Regra de aplicação: impede exibição de inativas if (!planta.ativo) { throw new Error("Esta planta está indisponível."); } return planta; } }

03. Camada de Infraestrutura

Tecnologias e Detalhes de Baixo Nível

A camada de infraestrutura dá suporte técnico às demais camadas. Ela contém as implementações físicas e ferramentas de terceiros.

Esta camada lida com:

  • Chamadas de rede (Fetch API, Axios, rotas).
  • Sistemas de persistência local (LocalStorage, Cookies, SessionStorage).
  • Injeção de dependências (containers de DI).
  • Mapeamento de payloads de API brutas em objetos de domínio.

DIP: Inversão de Dependências

Ao invés do domínio depender da infraestrutura, a infraestrutura depende do domínio ao implementar suas interfaces. Isso inverte a direção de dependência clássica.

Infra: Repositório HTTP

HttpPlantaRepository

Implementa o contrato IPlantaRepository. Efetua chamadas `fetch` à API REST e transforma a resposta JSON bruta em instâncias ricas da entidade Planta.

📂 src/core/infra/http/HttpPlantaRepository.ts

export class HttpPlantaRepository implements IPlantaRepository { constructor(private readonly apiUrl: string) {} public async getPlanta(slug: string): Promise<Planta> { const res = await fetch(`${this.apiUrl}/api/plantas/${slug}`); if (!res.ok) throw new Error("Erro ao obter planta"); const data = await res.json(); // Instanciação rica: mapeia JSON para Domínio return new Planta( data.id, data.slug, data.titulo, data.descricao, new Preco(Number(data.preco)), data.imagens, data.terreno_minimo_m2, data.ativo ); } }

04. Camada de Apresentação

Interface Visual e Interação Reativa

A camada de apresentação é o ponto de contato com o usuário final. Focada no design, layout e captura de interações.

Estruturação da Camada:

  • Hooks Customizados: Conectam a UI aos Casos de Uso.
  • Componentes Visuais: Estrutura limpa de exibição.
  • Roteamento: Captura parâmetros e renderiza as visualizações principais.

Separação de Lógica e Render

Os componentes do React não fazem chamadas HTTP nem validam regras diretamente. Eles recebem dados já processados do hook customizado e apenas renderizam a tela.

Apresentação: Hooks

usePlantaDetalhada

Hook customizado que busca a dependência do caso de uso pelo DI context, executa o fluxo e expõe o estado reativo de carregamento, erros e dados da planta.

📂 src/ui/hooks/usePlantaDetalhada.ts

export function usePlantaDetalhada(slug: string) { const { obterPlantaUseCase } = useDependencies(); const [planta, setPlanta] = useState<Planta | null>(null); const [loading, setLoading] = useState(true); const [error, setError] = useState<string | null>(null); useEffect(() => { obterPlantaUseCase.execute(slug) .then(res => setPlanta(res)) .catch(err => setError(err.message)) .finally(() => setLoading(false)); }, [slug, obterPlantaUseCase]); return { planta, loading, error }; }

05. Garantia de Qualidade

Pirâmide de Testes Automatizada

Um sistema robusto requer testes eficientes e adequados para cada tipo de código na aplicação.

Nossa Estratégia de Testes:

  • Testes Unitários: Focados no Domínio e nos Casos de Uso (sem navegador).
  • Testes de Integração de UI: Focados no comportamento do componente e interações do DOM.

Por que desacoplar ajuda a testar?

Como a lógica de negócio e os dados não estão misturados com requisições à rede HTTP ou hooks globais de tela, podemos testá-los instantaneamente no terminal usando implementações mockadas rápidas.

Testes: Nossas Ferramentas

Vitest & React Testing Library

A combinação perfeita para testar aplicações web modernas com altíssima performance.

Papéis e Diferenças:

  • Vitest: Test runner extremamente veloz baseado em Vite e ESM nativo. Substitui o Jest com as mesmas APIs familiares.
  • React Testing Library (RTL): Provê utilitários para interagir com o DOM virtual. Força boas práticas focando no comportamento do usuário final, em vez de detalhes de implementação interna.

Por que Vitest + RTL?

O Vitest elimina a necessidade de transpilação lenta de código TypeScript, rodando testes instantaneamente. A RTL garante testes de tela que não quebram ao refatorar a estrutura interna de tags HTML do componente.

O Fluxo Unidirecional de Dados

1. Interaction (UI)

O usuário clica no botão "Comprar Planta" no componente de página React.

2. Delegation (Hook)

O component chama o método do hook customizado useCheckout, que muda o estado local para loading.

3. Execution (Use Case)

O hook aciona o caso de uso CriarPedidoUseCase injetado via React Context.

4. Adaptation (Infra)

O caso de uso invoca o repositório HTTP da infraestrutura, que faz a chamada fetch para a API do backend.

5. Response & Update

O fluxo retorna instanciando entidades, o hook atualiza o estado com o preço formatado e a UI renderiza a tela.

Mapeamento da Estrutura Clean

Como a arquitetura se materializa no sistema de arquivos do Next.js de forma escalável:

frontend/src/ ├── core/ │ ├── domain/ ← Lógica de Negócio Pura │ │ ├── entities/ ← Ex: Planta.ts │ │ ├── value-objects/ ← Ex: Preco.ts │ │ ├── errors/ ← Ex: PrecoNegativoError.ts │ │ └── repositories/ ← Interfaces de Contrato │ ├── application/ ← Orquestração / Casos de Uso │ │ └── use-cases/ ← Ex: ObterPlantaDetalhadaUseCase.ts │ └── infra/ ← Implementações e Detalhes │ ├── http/ ← Repositório HTTP (fetch) │ ├── storage/ ← LocalStorageStorage.ts │ └── di/ ← Container Dependency Context └── ui/ ← Camada de Apresentação (React/Next) ├── components/ ← Componente PlantaDetalheView.tsx ├── hooks/ ← Hook usePlantaDetalhada.ts └── app/ ← Páginas de roteamento [slug]/page.tsx

Segurança em Refatorações

Se as regras do backend mudarem ou se a infraestrutura for alterada para outra biblioteca, a única pasta que sofre modificações é a core/infra/. Toda a interface visual (ui/) e a lógica pura (core/domain/) permanecem intocadas e seguras.
